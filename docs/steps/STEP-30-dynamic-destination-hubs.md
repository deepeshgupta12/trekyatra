# STEP 30 — Dynamic Destination Hubs

## Goal
Build programmatic destination hub pages (regional and seasonal) that auto-aggregate content from the pipeline. These are high-value SEO landing pages that scale without manual editing.

## Scope

### Regional hub pages (frontend)
- `/regions/[slug]` already exists (Step 18) but renders static data — wire to real CMS data
- Hub page pulls: all trek guides for this region, average difficulty, season range, featured image
- Related content: packing lists, permit pages, comparison pages for this region
- Schema markup: `TravelDestination` JSON-LD

### Seasonal landing pages (automated)
- `/seasons/[slug]` (already exists) — wire to real seasonal content
- `SeasonalContentAgent` (new): given a season slug, generates a seasonal overview page surfacing top treks, weather notes, packing priorities
- Output stored as a CMSPage with `page_type = seasonal_hub`
- Beat task: quarterly — re-generate seasonal hub pages (spring / summer / monsoon / winter)

### Auto-generated cluster landing pages
- `/trek-types/[slug]` (new route group): e.g., `/trek-types/high-altitude`, `/trek-types/weekend`
- Auto-generated from keyword cluster tags; content generated by ContentWritingAgent
- No separate DB table — stored as CMSPages with `page_type = cluster_hub`

### Admin UI
- `/admin/hubs` (new page): list of hub pages (regional + seasonal + cluster), last generated date, regenerate button per hub

### Backend
- `SeasonalContentAgent` new agent node
- `GET /api/v1/admin/hubs` — list hub pages by page_type
- `POST /api/v1/admin/hubs/{slug}/regenerate` — trigger content regeneration
- Celery beat: quarterly seasonal hub regeneration
